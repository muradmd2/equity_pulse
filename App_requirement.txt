Requirement Document: AI Financial Research App Using LangGraph
1. Goal
Build an educational AI financial research application to learn:
	• LangGraph
	• Agentic workflows
	• Multi-node graph orchestration
	• Parallel node execution
	• Tool-based financial data retrieval
	• LLM-based routing, summarization, and review
	• Streaming responses
	• LangSmith observability
	• Checkpointing with SQLite
The app should accept a user’s financial query, classify the query into one or more financial research categories, run the relevant category nodes in parallel, summarize the collected information, review the answer for missing pieces, optionally regenerate once, and return a source-cited final response.
This app is for learning and research only. It must not provide personalized investment advice.

2. Primary User Flow
	1. User enters a financial query.
	2. The graph receives the query.
	3. The first node classifies the query into one or more categories.
	4. The graph generates category-specific subqueries.
	5. Relevant category nodes run in parallel.
	6. Each category node retrieves data from the appropriate external source.
	7. Retrieved data is normalized into a common result format.
	8. The summary node combines all results into a concise response.
	9. The review node checks whether the answer fully addresses the original user query.
	10. If the review finds missing information and this is the first review attempt:
		○ Stream the message: Review failed. Regenerating the response.
		○ Rewrite the query.
		○ Send the rewritten query back to the classification node.
		○ Run the graph again for the missing categories only when possible.
	11. On the second summary pass, combine the first summary and second summary.
	12. Do not run the review node a second time.
	13. Return the final response with citations.

3. Query Categories
The classification node should identify one or more of the following categories:
	1. Company information
	2. Competitor news
	3. Sector news
	4. Financials
	5. Company news
	6. Current stock price
	7. Analyst ratings
	8. Latest report release
	9. Dow Jones index
	10. Technical analysis
The classifier must support multi-category queries.
Example:
User query:
	How is CRM stock doing today and what do analysts think about it?
Expected categories:
	• Current stock price
	• Analyst ratings
	• Company news, if the query asks “why” or “what happened today”

8. Graph Flow
Implement the graph as follows:
START
  -> classify_query_node
  -> build_subqueries_node
  -> dynamic parallel category nodes
  -> aggregate_results_node
  -> summary_node
  -> conditional:
       if review_attempted == false:
           review_node
       else:
           final_response_node
review_node
  -> conditional:
       if missing_info_found == true and review_attempted == false:
           rewrite_query_node
       else:
           final_response_node
rewrite_query_node
  -> classify_query_node
final_response_node
  -> END
Important:
	• The review node must run at most once.
	• Use a state field such as review_attempted: bool.
	• Use a state field such as iteration: int.
	• On regeneration, set:
		○ review_attempted = true
		○ iteration = 2
	• After the second summary, route directly to final_response_node.

9. Parallel Execution Requirement
After classification, the graph should route to all relevant category nodes.
Use dynamic routing so that only needed nodes execute.
Example:
User query:
Tell me about CRM stock, current price, recent news, financials, and analyst ratings.
The graph should run these nodes in parallel:
	• company_info_node
	• current_stock_price_node
	• company_news_node
	• financials_node
	• analyst_ratings_node
Do not run unrelated nodes such as Dow Jones or sector news unless classification says they are needed.

10. State Schema
Create a strongly typed state model.
Recommended state:
from typing import TypedDict, Annotated, Literal, Optional
import operator
class SourceCitation(TypedDict):
    source_id: str
    title: str
    url: str
    provider: str
    published_at: Optional[str]
    retrieved_at: str
    snippet: Optional[str]
class RetrievalResult(TypedDict):
    category: str
    subquery: str
    status: Literal["success", "partial", "failed"]
    data: dict
    summary: str
    citations: list[SourceCitation]
    errors: list[str]
class FinancialResearchState(TypedDict):
    original_query: str
    active_query: str
    rewritten_query: Optional[str]
detected_tickers: list[str]
    detected_companies: list[str]
categories: list[str]
    subqueries: dict[str, str]
retrieval_results: Annotated[list[RetrievalResult], operator.add]
    citations: Annotated[list[SourceCitation], operator.add]
    errors: Annotated[list[str], operator.add]
    warnings: Annotated[list[str], operator.add]
first_summary: Optional[str]
    second_summary: Optional[str]
    final_summary: Optional[str]
missing_info_found: bool
    missing_info: list[str]
review_attempted: bool
    iteration: int

11. Classification Node
The classify_query_node should:
	1. Accept original_query or active_query.
	2. Extract ticker symbols.
	3. Extract company names.
	4. Determine query categories.
	5. Return structured JSON.
Use OpenAI structured output with a Pydantic model.
Classification output schema:
class QueryClassification(BaseModel):
    detected_tickers: list[str]
    detected_companies: list[str]
    categories: list[
        Literal[
            "company_information",
            "competitor_news",
            "sector_news",
            "financials",
            "company_news",
            "current_stock_price",
            "analyst_ratings",
            "latest_report_release",
            "dow_jones_index",
            "technical_analysis",
        ]
    ]
    reasoning: str
Requirements:
	• If no ticker is found, attempt to infer ticker from company name.
	• If the company name is ambiguous, interrupt and ask for more information.
	• If the query is broad, select multiple relevant categories.
	• If the query asks “how is the stock doing,” include:
		○ current_stock_price
		○ company_news
		○ technical_analysis
	• If the query asks “is it a buy,” include:
		○ analyst_ratings
		○ financials
		○ company_news
		○ technical_analysis

    In the summary to the user:
	• Do not provide direct investment advice.
	• Do not provide factual information unless backed up by the retrieved information.
    . Do not generate more information than what is requested.  
    . Keep it concise and up to what is being requested in the initial query.
    . For providing news, provide only few lines of information and then provide a link to the news article.
    . For analyst rating, provide only rating unless query asks for detail rating information.
    . In financial analysis, be brief.
    . While providing earnings, emphasize on hit or miss.

12. Subquery Builder Node
The build_subqueries_node should create a concise search or retrieval query for each selected category.
Example output:
{
  "current_stock_price": "CRM current stock quote latest price",
  "company_news": "CRM Salesforce latest company news last 7 days",
  "financials": "CRM Salesforce latest quarterly financials revenue net income cash flow",
  "analyst_ratings": "CRM analyst ratings recommendation trends price target",

Each category node should receive only its own subquery and relevant ticker/company data.

Each category node must return a RetrievalResult.
All nodes must:
	1. Catch exceptions.
	2. Return partial results when possible.
	3. Include citations.
	4. Include provider name.
	5. Include retrieval timestamp.
	6. Avoid crashing the whole graph if one data source fails.
	7. Use fallback providers where appropriate.
	8. Normalize output into the shared RetrievalResult schema.

14. Aggregation Node
The aggregate_results_node should:
	1. Collect all category node outputs.
	2. Separate successful, partial, and failed results.
	3. Deduplicate citations.
	4. Deduplicate repeated facts.
	5. Preserve category labels.
	6. Preserve source metadata.
	7. Pass normalized data to the summary node.
If some nodes fail, continue with available results.

15. Summary Node
The summary_node should:
	1. Stream the answer progressively.
	2. Summarize all collected category outputs.
	3. Be concise but complete.
	4. Include citations inline or in a final source list.
	5. Mention data limitations.
	6. Avoid unsupported conclusions.
	7. Avoid personalized financial advice.
	8. Combine first and second summaries if regeneration occurred.
The summary should include these sections when relevant:
## Quick Answer
## Stock Snapshot
## Company Overview
## Recent News
## Financials
## Analyst View
## Technical Picture
## Sector / Competitor Context
## Latest SEC Filings
## Risks and Limitations
## Sources
Only include sections relevant to the user’s query.

16. Streaming Requirement
The app should not wait until the full graph finishes before showing output.
Implement streaming using LangGraph streaming.
The app should stream:
	1. Node progress updates
	2. Summary tokens
	3. Review failure message
	4. Final answer
Example streamed events:
Analyzing query...
Detected categories: current stock price, company news, analyst ratings
Fetching stock quote...
Searching recent company news...
Retrieving analyst ratings...
Summarizing results...
Review failed. Regenerating the response.
Fetching missing financials...
Finalizing response...
For the final response, stream the summary node’s LLM output token-by-token where possible.

17. Review Node
The review_node should compare:
	• original user query
	• detected categories
	• retrieved results
	• summary output
It should determine:
	1. Was the original query answered?
	2. Are any requested categories missing?
	3. Are citations missing?
	4. Are important ticker/company details missing?
	5. Did any failed node cause a major gap?
Review output schema:
class ReviewResult(BaseModel):
    missing_info_found: bool
    missing_info: list[str]
    rewritten_query: str | None
    reasoning: str
Rules:
	• If no major gap exists, go to final response.
	• If a major gap exists and review_attempted == false, route to rewrite node.
	• If review_attempted == true, skip review and go to final response.
	• Never allow infinite review loops.

18. Rewrite Query Node
The rewrite_query_node should:
	1. Use the original query.
	2. Use the review result.
	3. Generate a focused rewritten query for missing information only.
	4. Set review_attempted = true.
	5. Set iteration = 2.
	6. Preserve first-pass results and first summary.
	7. Stream:
Review failed. Regenerating the response.
Example:
Original query:
Tell me about CRM financials and analyst ratings.
First pass missed financials.
Rewritten query:
CRM latest quarterly financials revenue net income operating income cash flow SEC 10-Q

19. Final Response Node
The final_response_node should:
	1. Combine first and second summaries if regeneration occurred.
	2. Include source citations.
	3. Include warnings and limitations.
	4. Include failed-source notices if important.
	5. Include educational disclaimer.
Final disclaimer:
This is for educational research only and is not personalized financial advice.

20. Citation Requirements
Every external fact should be traceable to a source.
Each citation should include:
	• provider
	• source title
	• URL if available
	• published date if available
	• retrieved timestamp
	• snippet or short evidence text
Final response should include a Sources section.
Example:
## Sources
1. yfinance - CRM quote data - retrieved 2026-06-11 15:30 CT
2. SEC EDGAR - Salesforce 10-Q, filed 2026-xx-xx - URL
3. Tavily - Reuters article about Salesforce earnings - published 2026-xx-xx - URL
4. Finnhub - CRM recommendation trends - retrieved 2026-06-11 15:31 CT
Do not invent citations.
If a fact cannot be sourced, either omit it or label it as unavailable.

21. Error Handling
The app should handle:
	1. Invalid ticker
	2. Ambiguous company name
	3. API key missing
	4. API rate limit
	5. Empty Tavily response
	6. yfinance missing fields
	7. SEC ticker-to-CIK failure
	8. Finnhub quota failure
	9. Network timeout
	10. Partial data availability
	11. LLM classification failure
	12. LLM malformed JSON response
Each node should return a partial RetrievalResult instead of crashing the graph where possible.
Example:
{
  "category": "analyst_ratings",
  "status": "failed",
  "data": {},
  "summary": "Analyst ratings could not be retrieved because Finnhub returned an error.",
  "citations": [],
  "errors": ["Finnhub API rate limit exceeded"]
}

22. Guardrails
The app must:
	1. Not provide personalized investment advice.
	2. Not say “you should buy” or “you should sell.”
	3. Use language like:
		○ “Analysts currently rate...”
		○ “The data suggests...”
		○ “Risks include...”
		○ “This may be worth researching further...”
	4. Clearly separate facts from interpretation.
	5. Mention data freshness.
	6. Mention when market data may be delayed.
	7. Avoid hallucinating unavailable financial metrics.
	8. Avoid using unsourced claims.
	9. Not compare stocks as a recommendation unless the user explicitly asks for a comparison.
	10. Include an educational disclaimer.

32. Important Implementation Notes
	• Use structured LLM outputs wherever possible.
	• Use deterministic temperature for classification and review.
	• Prefer temperature=0 for routing and review logic.
	• Use a slightly higher temperature only for final natural-language summarization if desired.
	• Keep raw retrieved data separate from LLM-generated summaries.
	• Never let the LLM invent financial data.
	• Every node should return structured data.
	• Every external source response should be traceable.
	• Keep the system modular so new providers can be added later.
	• Add comments explaining LangGraph concepts because this is a learning app.
    • Keep the string literal and LLM instructions in a separate file.
    • Use constants where you can.
